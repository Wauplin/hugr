# An agent binary from Python

This guide takes an agent crate you already built, using `examples/huglet-docs`, and ships it to Python users in three ways. The options are a **typed wheel** generated by `huggr build --surface python`, a plain **subprocess** that speaks the JSON ask/answer contract, and an **MCP server** via `--mcp-serve`.

All three wrap the same bundle. This guide explains what each surface provides and when to use it. For errors as answers, mandatory `AnswerMeta`, and trace lineage, see [the Ask and Answer contract](../agents.md#the-ask-and-answer-contract).

## What "shipping to Python" means here

A Huggr agent is defined once, in its crate folder (`huggr.toml` + `SYSTEM.md` + an optional Rust response contract). The Python wheel is *generated from* that definition; you never write Python to ship it. This is deliberately distinct from guide 05, where you *define* an agent in Python: here Python is only a consumer of an already-built agent.

## Prerequisites

- A working `huggr` CLI (`cargo install --path crates/huggr-toolkit` from the repo root, or your existing install).
- [maturin](https://www.maturin.rs) on your PATH (`pip install maturin`) as the wheel builder.
- An API key for the agent's provider. `huglet-docs` reads `HUGGR_API_KEY` (see its `huggr.toml`).

## Surface 1: the typed wheel

From the repo root:

```bash
huggr build examples/huglet-docs --surface python --release
```

This first builds the ordinary CLI binary. The `python` surface runs that binary with `--config --json` to read the compiled response schema, so the Python types come from the same schemars output seen by the model. It then creates a maturin "mixed" project under `examples/huglet-docs/dist/huglet_docs-python/` and builds a wheel:

```text
huglet_docs-python/
  Cargo.toml           # cdylib `_native`, links huggr-toolkit + the agent crate
  pyproject.toml       # maturin backend, module-name = huglet_docs._native
  bundle.bin           # the embedded agent bundle (same bytes as the CLI shim)
  src/lib.rs           # PyO3: ask_json(...) -> Answer JSON, in-process
  python/huglet_docs/
    __init__.py        # typed ask(...) -> Answer
    _models.py         # generated dataclasses: DocsResponse, Answer, AnswerMeta, BlobHandle
    py.typed           # PEP 561 marker
```

The build prints the wheel path; install it:

```bash
pip install examples/huglet-docs/dist/huglet_docs-python/target/wheels/huglet_docs-*.whl
```

The module name is the agent name with dashes mapped to underscores (`huglet-docs` → `huglet_docs`). Use it:

```python
import huglet_docs

answer = huglet_docs.ask("./docs", "How do I resume a trace?")
if answer.ok:
    print(answer.response)          # a typed DocsResponse dataclass, not a dict
else:
    print("error:", answer.error)
print(answer.trace_id, answer.metadata.cost_micro_usd)
```

Three things to notice:

- **Declared runtime args become typed parameters.** `huglet-docs` declares a positional, required `docs_path` runtime arg in its manifest, so the generated `ask()` signature is `ask(docs_path, question, *, trace_id=None, blobs=None, skills=None, extra=None)`; positional args lead, before the question, mirroring the CLI surface exactly. `skills` accepts caller-local standard `SKILL.md` folder paths for that invocation.
- **The typing is strict but there is no second validator.** Rust already casts the model's final JSON into the agent's response type before it reaches `Answer.response`; `_models.py` only deserializes that already-valid JSON into dataclasses.
- **It runs in-process.** The PyO3 bridge embeds the bundle and drives the real runtime inside your Python process; no subprocess, no serialization boundary beyond one JSON string. Traces still land under `~/.huggr/huglet-docs/` like every other surface.

Resume works the same as everywhere else; pass a previous answer's `trace_id` and you get a *new* trace with `depends_on` set:

```python
follow_up = huglet_docs.ask("./docs", "And how do I fork one?", trace_id=answer.trace_id)
```

## Surface 2: the binary as a subprocess

If you do not want to build or install a wheel, for example from a language without bindings or when keeping the agent fully out of process, every built binary already speaks the JSON contract: one `Answer` object on stdout, diagnostics on stderr, exit 0 even for error answers (errors are answers).

```bash
huggr build examples/huglet-docs --release   # the default `cli` surface
```

```python
import json, subprocess

proc = subprocess.run(
    ["./examples/huglet-docs/dist/huglet-docs", "./docs", "How do I resume a trace?", "--json"],
    capture_output=True, text=True,
)
answer = json.loads(proc.stdout)
print(answer["status"], answer["trace_id"], answer["metadata"]["cost_micro_usd"])
```

This approach returns the response as a plain dict instead of generated types, but provides full process isolation. Huggr uses the same mechanism when one agent grants another as a tool. The binary also has audit subcommands worth knowing: `--describe` (the agent card), `--config` (the parsed manifest + response schema), `--traces`, and `--stats`.

## Surface 3: `--mcp-serve`

The same binary can serve itself as an MCP server over stdio, so any MCP client (an editor, another agent framework, a Python MCP SDK) can call it as a tool without knowing anything about Huggr:

```bash
./examples/huglet-docs/dist/huglet-docs ./docs --mcp-serve
```

Register that command line in your MCP client's config and the agent shows up as a callable tool; each call is one ask, and traces still accumulate under `~/.huggr/huglet-docs/`.

## Choosing between them

- **Wheel:** you control the Python codebase and want typed responses, IDE completion, and in-process speed.
- **Subprocess:** you want isolation, a non-Python caller, or the smallest possible integration surface (spawn + parse JSON).
- **MCP:** the caller is an existing MCP-speaking host and you don't want any bespoke glue.

Whichever you pick, the agent's definition, privileges, traces, and cost accounting are identical; the surface is only packaging.
